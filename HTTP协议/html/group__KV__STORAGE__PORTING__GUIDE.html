Background

The KV (Key-Value) Storage HAL is used by ACS modules as a way of persistently storing/retrieving data on target platform.
Since the KV Storage HAL is a set of platform dependent APIs, this document describes a list of porting requirements, implementation guidance as well as API and test dependencies.

Porting prerequisites

  1. A storage or filesystem supporting read/write operations.
  2. The storage or filesystem should:
    1) Have pre-created data groups.
    2) Be able to create data groups on demand.

Rules and limits

  1. KV Storage must be able to store single binary value size up to ACE_KV_STORAGE_HAL_MAX_VAL_LEN.
  2. KV Storage keys have the format of X.Y where X is the group name and Y is the key name.
    1) The length limit of KV Storage group name is ACE_KV_STORAGE_HAL_MAX_GROUP_LEN.
    2) The length limit of KV Storage key name is ACE_KV_STORAGE_HAL_MAX_KEY_LEN.
    3) If no group name is specified, the assumption is that the key Y is to be accessed from default group (pre-defined).
    4) If X.Y key format is provided to KV Storage HAL APIs, it assumes that user would like to access the key Y in group X.
    5) If the group X is not available, then it should return ACE_STATUS_NOT_FOUND.
    • Note that, the entire string after the first '.' should be treated as the key name.
    • For example "somegroup.somekey_subkey", "somegroup" is the group name and "somekey_subkey" should be treated as the key name.
    • No "."(dot) is allowed in group key and key name.
  3. ACE_KVS_DS_DEFAULT_GROUP option needs to be configured to define the default group name.
  4. KV Storage set API aceKeyValueDsHal_set will overwrite the existing value.
  5. aceKeyValueDsHal_setWithEncryption API store the data with hardware-backed encryption.
    1) Data must be protected at rest through encryption with the encryption key not accessible widely, such as a hardware crypto engine or Trusted Execution Environment (TEE).
    2) Authenticated Encryption such as AES-GCM must also be used.
    • Example: Encrypt data using AES-GCM and the AES key is protected by a hardware-backed PKCS#11 implementation.

Implementation details

The following set of APIs are used to access KV Storage:

  • aceKeyValueDsHal_db_init
  • aceKeyValueDsHal_set
  • aceKeyValueDsHal_setWithEncryption
  • aceKeyValueDsHal_remove
  • aceKeyValueDsHal_get
  • aceKeyValueDsHal_getValueSize
  • aceKeyValueDsHal_eraseGroup
  • aceKeyValueDsHal_listKeys

Shared library name: libacehal_kv_storage

For detailed information on individual APIs, please refer to the API documentation.

Testing

Run KV Storage HAL test to validate the KV Storage HAL API implementation.
The implementation must pass all KV Storage HAL tests.

Performance and Stability

  1. The KV Storage access latency should meet the system or application level requirements of the product.
  2. Data integrity must be guaranteed in situations like lost power in the middle of writing to KV Storage (eg, aceKeyValueDsHal_set).
  3. A mechanism to make sure data is correctly read/written is recommended (eg, CRC checksum).
  4. “Flash wear-out” should be taken into consideration.
