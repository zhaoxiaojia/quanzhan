Overview

ACS unifies the API integration layer across Amazon device SDKs and provides pre-validated, memory-optimized components for common functionalities such as connectivity, multimedia, and logging. ACS is built on a porting layer, the Device Porting Kit (DPK), which abstracts the hardware and operating system. ACS includes extensive test suites to verify and debug an ACS implementation. In addition to the portability, ACS is designed to provide a highly configurable software distribution model for constituent software components.

Benefits

Launch Products Faster

ACS accelerates integration of Amazon Device SDKs and their software dependencies by only requiring one single effort.

Reduce Development Costs

ACS reduces software development and maintenance costs by using pre-built software components that are portable and memory optimized.

Build High Quality Devices

Use ACS to build high quality and reliable experiences with pre-hardened and pre-validated software components that have been used in Amazon devices.

Keep Devices Secure

ACS keeps your device fleet secure with frequent feature and security updates.

Architecture

First, ACS provides a single, common API integration layer for integrating multiple Amazon SDKs. Once the API integration layer is implemented, you can re-use it with any Amazon device SDK supported by ACS. For example, if you build a smart home device using ACS and the AVS SDK, you can easily add FFS and AWS IoT SDKs in the future. Second, ACS includes pre-validated and memory-optimized components that implement the common API integration layer components such as connectivity, logging, metrics, audio management, authentication, and key-value store. Third, ACS components are portable to any operating system using the DPK, and are already available on FreeRTOS and Embedded Linux. Fourth, ACS includes extensive test suites that allow you to verify and debug your ACS implementation quickly and independently. The diagram below shows the ACS high-level architecture.

Device Porting Kit (DPK)

Device makers need to port the DPK APIs on to their specific platforms. The DPK includes an OS Abstraction Layer (OSAL) and Hardware Abstraction Layer (HAL). The OSAL provides OS primitive APIs (e.g., thread creation, synchronization, timers) that can be mapped to POSIX and non-POSIX compliant operating systems (a:FreeRTOS). To make the porting easy for POSIX compliant OSes, the OSAL APIs are mapped to an open source library such as glibc. For Amazon FreeRTOS, we created a direct mapping to the FreeRTOS APIs. The ACS HAL APIs are high-level abstractions for device-specific peripherals (e.g., LED, button, WiFi, BT). To ease the porting effort for device makers, ACS offers reference implementation of HALs on Android, Linux, and FreeRTOS. The device maker needs to customize these reference implementations to match their board/device-specific changes.

In addition to these APIs, ACS includes a number of building blocks that help software modules to work efficiently on different operating systems. For example, ACS offers a "dispatcher" that can mimic an event-driven execution model for a single core MCU with limited memory. The dispatcher has a configurability option to change the execution model from event-driven model to concurrent model. This helps the software component using the same API to work efficiently across a resource-constrained MCU and a full-fledged SoC. ACS abstracts IPC complexity, life cycle management, and feature configurability for module developers. These features free middleware developers from worrying about the underlying OS-specific implementations.

Distribution and Updatability

ACS is designed to handle modular distribution and updates for any of its constituent middleware components. SDK and component owners will be able to use a self-service workflow ("ACS Desk") to handle configuration, vending, and distribution of software. For example, AVS or ASDK can use this tool to vend pre-configured software stacks and they can also determine how much customization an end customer can make (i.e., which components can be added/removed and which can be vended independently). The ACS Desk automatically builds these packages from the same code line as our internal products and the package is customized to meet the IDE/toolchain requirements for the customer. A set of micro services are built on AWS resources that will interact with the backend SCM tools for software vending.

Testability

ACS offers a multi-level validation suite to ensure end-to-end functional maturity and performance of the whole stack including vendor software. The ACS software stack includes middleware components from different vendor software, open source packages, and toolchains. The ACS Validation Test Suite (VTS) covers all the components mentioned previously. Refer to Testing for additional details.

Use Cases

ACS use cases are dependent on the reference platform being used. Typical use cases are listed below.

WiFi Simple Setup

Add WiFi Simple Setup to your Alexa built-in product and allow it to be discovered and set up through the Alexa Mobile Companion app. ACS provides an optional connectivity (Bluetooth and WiFi) stack and middleware components, if needed.

Creating an Alexa Gadget endpoint

Get Amazon FreeRTOS, a bluetooth stack, and all the device software components needed to create a product that connects with and extends Echo devices using the Alexa Mobile Accessories (AMA) protocol.

Making your hardware Amazon-ready

Add your hardware platform to the list of qualified platforms by implementing the ACS Device Porting Kit. ACS provides reference implementations for Amazon FreeRTOS and Embedded Linux to make it easy to get started.

Testing

ACS offers a multi-level validation suite to ensure end-to-end functional maturity and performance of the whole stack including vendor software. The ACS software stack includes middleware components from different vendor software, open source packages, and toolchains. The ACS Validation Test Suite (VTS) has the following core components:

  • Device Porting Kit Test Suite (DTS) : DTS consists of functional API tests for OSAL and HAL layer components. It contains API tests which are developed using ACS Testing Framework (ATF) and packaged onto standalone CLI executable generated by the build system. Execution of the DTS suite validates DPK implementation of an ACS-fied product.
  • ACS API Layer Test Suite (MTS) : MTS validates functionality of Middleware layer components. It consists of API tests which validate the functionality of Middleware APIs. Most of the API tests would be C-based test which would be developed using the ACS Testing Framework. Tests can be executed on the device by running CLI executable for each component on the device.
  • SDK Layer Test Suite (SDKTS) : SDKTS validates functionality of specific Amazon technology components (e.g., AVS, FFS, ASDK, etc.). Since SDK components are one of the core features a customer would put on their 3P devices, the test suite will help validate functionality of the SDK component from ACS point of view. Existing SDK components (e.g., AVS) test suites can also be integrated on SDKTS to provide a unified testing experience. Tests will be developed, owned and maintained by SDK provider.
  • APP Layer Test Suite (ATS) : ATS tests validate ACS sample apps (e.g., Spencer Sample APP, OOBE sample APP) functionality and performance.

Support

The ACS Portal provides a single window interface to deploy ACS in a self-service manner. Through the portal, users provide the desired platform configuration, choose the required ACS feature set, and download the ACS software along with tools and instructions to build, test and certify. Users will also have access to support artifacts such as documentation and FAQs, including build and troubleshooting tools required for integrating ACS.
